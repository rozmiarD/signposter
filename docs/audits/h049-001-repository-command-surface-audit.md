## Current Architectural Direction

The current repository is no longer a skeleton. The strongest next direction is not a new execution backend. The highest-value stage is control-plane maturity:

- make planner, scheduler, lifecycle, and orchestrator agree on DAG state;
- make stuck states visible and recoverable;
- reduce brittle gate and evidence heuristics;
- keep GitHub comments bounded and safe;
- harden review, merge, integration, cleanup, and no-op paths;
- improve token efficiency through prompt budgeting and conservative router policy;
- update docs to match implemented code.

## Planner and Scheduler Findings

The H049 manifest has 80 issues and the planner selected H049-001 / issue #208 as the next task.

Observed issue:

- `signposter planner run --manifest /tmp/signposter-h049-manifest.json --sync-github --dry-run` reported `ready: 80`.
- The same output correctly reported `79 task(s) waiting for dependencies`.

This is a real status-count clarity bug. It should be handled in the H049 planner/DAG workstream because it can mislead operators about dependency readiness.

Observed source-of-truth tension:

- Planner manifests preserve task-key dependencies.
- GitHub issue bodies generated by planner do not yet expose dependency metadata in a way that scheduler graph can reliably parse as GitHub issue references.
- The scheduler can select global `state:ready` issues outside the active roadmap unless the operator uses the intended manifest.
