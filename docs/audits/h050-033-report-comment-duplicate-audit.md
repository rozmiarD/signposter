# H050-033 Report Comment Duplicate Behavior Audit

## Scope

This audit covers what Signposter should do when `signposter report --apply` is
run more than once for the same issue and worker summary.

Audited surfaces:

- `src/signposter/report.py` comment formatting and posting;
- report comments generated by worker summary artifacts;
- local raw and summary artifact references;
- GitHub issue comment mutation boundaries;
- operator interpretation of repeated evidence comments.

Non-goals:

- no runtime behavior change;
- no report comment upsert implementation;
- no GitHub comment deletion or editing;
- no issue closure behavior change;
- no unrelated report formatting change.

## Current Behavior

`signposter report --apply` posts a new bounded GitHub issue comment every time
it is run. The command does not search for an existing Signposter report comment,
does not edit an existing comment, and does not delete older comments.

This is consistent with the current safety model:

1. Report comments are append-only evidence events.
2. Raw logs remain local; GitHub receives only bounded summaries and excerpts.
3. Issue closure is owned by integration, not report.
4. Re-running report is visible to the operator as a repeated evidence event.
5. No hidden comment mutation occurs without an explicit operator-visible plan.

## Duplicate Behavior Options

### Append Explicit Evidence Comments

Appending a new comment for every report apply keeps the audit trail simple.
Each report mutation is visible in chronological order, and GitHub preserves
who posted it and when.

Strengths:

- no hidden update semantics;
- preserves every report attempt;
- works with existing dry-run/apply boundaries;
- does not require matching brittle prior comment text;
- keeps recovery simple after interrupted runs.

Weaknesses:

- repeated report applies can add duplicate-looking comments;
- noisy issues may require more operator scanning;
- the latest summary is not singled out automatically.

### Update The Previous Signposter Report Comment

Updating an existing comment would reduce visual noise, but it changes the
meaning of report apply from "post this evidence event" to "find and mutate an
earlier evidence event." That requires new safety checks.

Required safeguards before implementation:

- dry-run output must show the exact target comment id;
- apply must edit only comments authored by the expected Signposter identity;
- matching must use a stable marker or metadata, not loose text search;
- the old and new bounded excerpts must remain locally auditable;
- failure to identify exactly one safe target must block, not append silently.

### Upsert Latest Report Comment

An upsert mode could be useful for noisy automation loops, but it must be an
explicit mode such as `--mode append|update-latest` or a separate command. It
should not become the hidden default for `report --apply`.

Required safeguards:

- default mode remains append-only;
- dry-run explains whether it would append or edit;
- update mode blocks on missing, multiple, or untrusted prior comments;
- edited comments must retain enough evidence that the operator understands the
  current artifact path, validation result, and issue state.

## Recommendation

Keep the current default behavior: repeated `signposter report --apply` should
append explicit bounded evidence comments.

Rationale:

- append-only comments match the current audit-first lifecycle;
- repeated report applies are safer when visible than when silently coalesced;
- integration still owns issue closure, so duplicate report comments do not
  change terminal state;
- the current implementation already keeps comments bounded and localizes raw
  artifacts;
- update/upsert requires a separate scoped implementation with stronger tests.

If operator noise becomes a practical problem, add a future explicit update mode
instead of changing the default. That task should be separate from this audit.

## Required Future Tests For Update Mode

If update/upsert is later implemented, test these cases before enabling apply:

- dry-run with no matching prior report comment plans an append or blocks
  according to the selected mode;
- dry-run with exactly one trusted prior report comment shows the comment id;
- dry-run with multiple matching comments blocks and does not guess;
- apply edits only a trusted Signposter-authored comment;
- apply never edits user comments or reviewer comments;
- unsafe body redaction still happens before posting or editing;
- auto-close keywords still block before any GitHub mutation.

## Operator Guidance

Current interpretation of repeated report comments:

- multiple Signposter runner report comments are expected append-only evidence;
- the latest comment should be treated as the newest report event;
- raw artifacts remain local even when report comments repeat;
- duplicate-looking comments do not imply issue completion;
- issue completion still requires gate pass, `complete`, PR/review/merge,
  integration, and cleanup.
